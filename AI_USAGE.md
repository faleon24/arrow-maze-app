## Entry 1 — App Phase 0 hardening: audit fixes and API polish
**Date**: 2026-07-11
**Tool**: Claude Opus 4.8 (via claude.ai)
**Task**: Before starting the v2 breaking-change work (arrow-path domain, neon render, Clean Architecture reshuffle), the Flutter app needed a sanitization pass to close every open finding in AUDIT.md §14 and eliminate cross-cutting smells that would only get worse as the surface grew. Ships seven sub-blocks plus one fix commit for a VS Code paste-fantasma incident, and closes Phase 0.B of PLAN-MASTER end to end.
**Prompt (paraphrased)**: I asked Claude to work in the smallest verifiable sub-blocks possible — one audit finding per commit, each with its own `flutter analyze` / `flutter test` / commit cycle — mirroring the cadence that had just closed Phase 0.A on the backend. Strategic framing: v2 is heavy and cross-repo, so the launchpad had to be clean before we touched it. Claude's role each round was to propose the exact minimal edit, wait for confirmation of a green analyzer plus tests, and commit before moving on.
**Result**: One deletion, three feature refactors, four small fixes, one style pass — plus one paste-fantasma recovery:
- **0.B.1 — Remove orphaned template test**: `test/widget_test.dart` referenced a `MyApp` class that never existed (leftover from `flutter create`). Deletion made `flutter analyze` clean for the first time in the project.
- **0.B.2 — Centralize base URL and add HTTP timeouts**: introduced `lib/data/api/api_config.dart` as the single source of truth for backend URL (via `String.fromEnvironment('API_URL', ...)`) and request timeout (10 s). All three API classes (Level/Auth/Progress) now share the constants; each `http.get/post` is wrapped in `.timeout(ApiConfig.requestTimeout)`. Killed three copies of a hard-coded `http://localhost:3000/api` string that would have drifted the moment a staging backend appeared.
- **Fix — restore AuthApi class after failed paste**: while applying the three API replacements for 0.B.2, VS Code's paste ate one of them and left `auth_api.dart` containing `ProgressApi`'s body. The analyzer complained that `AuthApi` was undefined in two screens. Recovered by clearing the file in VS Code, verifying `wc -l` returned 0, and re-pasting the correct content. Documented as a distinct `fix(data): restore AuthApi class dropped by failed paste` so the paste-fantasma bug shows in the log rather than being hidden inside the refactor.
- **0.B.3 — Typed API exceptions**: added `lib/data/api/api_exception.dart` with `ApiException(message, statusCode)` and `UnauthorizedException extends ApiException`, plus a factory `ApiException.fromResponse(http.Response)` that reads the backend's `message` field and routes 401s to the Unauthorized subclass. Removed three `e.toString().replaceFirst('Exception: ', '')` code smells across `login_screen`, `register_screen` and `game_screen`. Each API method now throws `ApiException.fromResponse(response)` on any non-success and `UnauthorizedException` on a missing local token.
- **0.B.4 — Token expiry + global 401 sign-out**: extended `AuthStorage` to persist `expiresAt` alongside the token (rename `saveToken`/`clearToken` → `saveSession`/`clearSession` for semantic clarity). `AuthGate` now validates expiry on launch and clears the session if the token is locally stale, so no doomed request fires. Added `lib/presentation/auth_guard.dart` with a `GlobalKey<NavigatorState>` and a static `AuthGuard.signOut()` that clears storage and pushes `LoginScreen` as the sole route — callable from any catch of `UnauthorizedException` regardless of `BuildContext`. `MaterialApp.navigatorKey` in `main.dart` wires the key so the reach is truly global. Only `game_screen` (the one authenticated call site so far) catches `UnauthorizedException` explicitly; login and register keep the generic `ApiException` catch because for them a 401 means "wrong credentials", not "session expired".
- **0.B.5a — Cancelable red-flash timer**: replaced `Future.delayed` with a nullable `Timer? _flashTimer`, cancel-then-restart on each blocked tap, cancel-on-dispose. Prevents overlapping callbacks from stomping each other's `setState` after fast double-taps.
- **0.B.5b — PopScope on end-of-game dialog**: wrapped the `AlertDialog` in `PopScope(canPop: false, ...)` so the Android system back button cannot pop the dialog and leave the player on a frozen board.
- **0.B.5c — Guard against empty-board level**: `_startSession` now schedules a `WidgetsBinding.instance.addPostFrameCallback` that fires `_submitAndShowWin()` if `_session.isCleared` right from the start (0-arrow board, or one where every arrow starts free). Without this, the win path would only run on the next tap — which for an already-cleared board never comes.
- **0.B.5d — Parallel catalog + progress fetch**: `LevelsScreen._load()` now fires the two futures concurrently and awaits both, using `.catchError((Object _) => <String, int>{})` on the progress side so a 401 or offline event still shows the catalog with empty stars. Roughly halves perceived load time.
- **0.B.6 — Remove dead code**: deleted `board_widget.dart` (unused), stripped `typeLabel` from the Cell hierarchy (unused getter, five copies), removed `StartCell` and `ExitCell` classes plus their two branches in `CellFactory`. Kept `WallCell` and `canTraverse` with docstring notes explaining that Fase 4 (v2 arrow-path model) revives them for ray-blocking.
- **0.B.7 — dart format lib test**: single-command style pass over the whole tree.

Final baseline after the block: `flutter analyze` reports 0 issues; `flutter test` runs 10 domain tests, all green. Nine commits shipped, all Conventional, all pushed to `origin/main`.
**Files affected**:
- `test/widget_test.dart` (deleted)
- `lib/data/api/api_config.dart` (new), `lib/data/api/api_exception.dart` (new)
- `lib/data/api/level_api.dart`, `lib/data/api/auth_api.dart`, `lib/data/api/progress_api.dart` (all rewritten to use `ApiConfig` and throw typed exceptions)
- `lib/data/auth_storage.dart` (expiry persistence + method rename)
- `lib/presentation/auth_guard.dart` (new)
- `lib/presentation/screens/auth_gate.dart` (validate expiry on launch)
- `lib/presentation/screens/login_screen.dart`, `register_screen.dart` (use `saveSession`, add `ApiException` import, kill `.replaceFirst`)
- `lib/presentation/screens/game_screen.dart` (cancelable timer + PopScope + empty-board guard + `UnauthorizedException` catch that calls `AuthGuard.signOut`)
- `lib/presentation/screens/levels_screen.dart` (parallel `_load`, adopt `clearSession` in logout button)
- `lib/main.dart` (wire `AuthGuard.navigatorKey`)
- `lib/domain/models/cell.dart`, `lib/domain/models/cell_factory.dart` (dead code removed)
- `lib/presentation/widgets/board_widget.dart` (deleted)
**Modifications made by the developer**:
- Enforced sub-block cadence: one edit, one analyzer+tests cycle, one commit, no batching until proven necessary. Nine commits, all Conventional Commits, all pushed granularly.
- Diagnosed a VS Code paste-fantasma bug during 0.B.2 where `auth_api.dart` was silently overwritten with `progress_api.dart`'s body. When Claude's first fallback (`cat > ... << 'EOF'`) also failed because the terminal was inside an unclosed heredoc, insisted on an empty-then-paste recipe: `Cmd+A → Delete → Cmd+S`, verify `wc -l == 0`, then paste. That two-step protocol became the new anti-fantasma default.
- When the `clearToken` → `clearSession` rename broke `levels_screen.dart` (which Claude had not asked to see), applied the fix and folded it into the same 0.B.4 commit as the atomic semantic unit rather than a follow-on afterthought. Trade-off consciously accepted: no legacy method wrappers to keep the API clean, at the cost of a call-site sweep.
- Verified each phase against real output before advancing: `flutter analyze`, `flutter test`, `grep -rn "replaceFirst" lib/presentation` after 0.B.3 to prove the smell was gone, `head -3` on every rewritten file after every paste to catch paste-fantasma before it hit a commit.
**Lessons learned**:
- **Paste-fantasma protocol**: for any multi-file paste of similar content (three APIs in a row, four cross-linked screens), the safe recipe is empty-first-verify-then-paste. `wc -l` on the empty file is the checkpoint that separates a good paste from a silent overwrite. `head -3` after the paste is the second checkpoint. Neither takes more than a second and both would have caught the 0.B.2 bug at the commit boundary instead of the analyzer boundary.
- **Renames without wrappers are cheap up front, expensive by one call site**. Killing `saveToken`/`clearToken` in favor of `saveSession`/`clearSession` was the semantically clean move, but the follow-through cost was one broken call site in a screen Claude had not asked about. Rule going forward: after a public method rename, `grep -rn "oldMethod" lib` before running tests — takes two seconds, avoids the "one more file" surprise.
- **Global side-effects need a global handle**. The `AuthGuard.navigatorKey` pattern is not just an FYI: it is the difference between "any authenticated future can force a sign-out" and "every catch block has to plumb a `BuildContext` down through async boundaries". Wire the key at MaterialApp once; every call site becomes a one-liner `on UnauthorizedException catch (_) => AuthGuard.signOut()`.
- **The dead-code pass is easier when it comes last**. Deleting `StartCell`/`ExitCell` and `typeLabel` while the earlier sub-blocks were changing the API surface would have generated conflicts; running it after 0.B.5 meant a `grep` of six symbols told the whole story of what to delete and what to keep. Fase 4 will revive `WallCell` — the docstring note is the reason a future teammate does not delete it again in six months.
- **The paste-fantasma is a two-tool bug, not a VS Code bug**. When the terminal heredoc also failed (an unclosed heredoc consumed the whole conversation buffer as input), the workaround became a **third** tool: VS Code with an empty-first invariant. Any long session should keep all three routes rehearsed; whichever tool bites first, the next is one command away.

## Entry 2 — App refactor to hexagonal architecture

**Date**: 2026-07-13
**Tool**: Claude Opus 4.7 (via Cowork desktop app)
**Task**: The app entered session 10 with a three-layer (`data/`, `domain/`, `presentation/`) structure that had accumulated coupling: screens instantiated adapters directly, no application layer existed, ports were absent, tests only covered domain models. Backend was already hexagonal; the app was the architectural asymmetry the professor would grade against. With Friday's deadline looming and the rubric weighting architecture, the highest-ROI move was a full refactor to hexagonal — ports, adapters, DI, application use cases — done in atomic commits with tests green at every step. Shipped as a nine-commit branch (`refactor/hex-architecture`) that merged to `main` after every commit was verified.

**Prompt (paraphrased)**: I asked Claude to design a refactor plan before touching code, capture it in a persistent `report.txt` for context, then execute in surgical sub-commits with per-commit `flutter analyze` + `flutter test` gates. Priority: the professor's grade on architecture. Cut anything with marginal ROI (F7 game-logic extraction, F8 sealed Failure hierarchy) if the four visible rubric items — ports, adapters, DI, application layer — were satisfied.

**Result**: Nine commits, 47 files touched at merge, tests moved from 60 to 78 green, 0 analyzer issues at every checkpoint.

- **F0 — Preparation**: added `get_it: ^7.7.0` to `pubspec.yaml`, killed two `unused_field` warnings in `cell_widget.dart` (`_background`, `_gridLine`). Branch cut from `main` after baseline confirmation.
- **F1 — Scaffolding (folded into F2)**: skipped as a standalone commit because empty `.gitkeep` folders are git-noise. Folder tree materialised naturally when F2 landed real files. Deviation from the plan documented in the sequence.
- **F2 — Domain ports and entities**: added `IAuthRepository`, `IAuthTokenStorage`, `ILevelRepository`, `IProgressRepository` in `lib/domain/ports/`, plus new domain entities `Level` and `UserSession` so the ports could return domain shapes instead of the transport DTOs the API layer had been leaking upward. All framework-agnostic — grep confirmed zero flutter/http/shared_preferences imports in `lib/domain/`. Pre-existing debt noted but not scope-crept: `arrow_color.dart` still imports `package:flutter/painting.dart`.
- **F3a — Scaffold hexagonal adapters**: created `lib/infrastructure/adapters/{http,local}/` and `lib/infrastructure/dto/` with fresh files that `implements` each port. `AuthResponseDto` and `LevelDto` gained `toDomain()` mappers projecting into the new domain entities. Old `data/api/*` classes stayed in place — no consumers touched yet, tests still green because nothing referenced the new adapters.
- **F3b — Presentation consumes ports**: rewrote `auth_gate`, `auth_guard`, `login_screen`, `register_screen`, `levels_screen`, `game_screen` and moved `test/data/dev_level_source_test.dart` → `test/infrastructure/adapters/local/dev_level_adapter_test.dart` via `git mv` to preserve history. Screens now type against interfaces — `AuthApi()` → `AuthHttpAdapter()`, `AuthStorage()` → `SharedPrefsTokenStorage()`. Sanity grep of `import.*data/` under `lib/presentation` returned empty. `ProgressHttpAdapter` gained a constructor-injected `IAuthTokenStorage` (the old `ProgressApi` self-instantiated it).
- **F3c — Delete legacy data folder**: `git rm -r lib/data/`. Nine files removed, 368 lines deleted. Tests green after removal because F3b already rewired every consumer.
- **F4 — DI container with `get_it`**: introduced `lib/core/di/service_locator.dart` with a `setupDI()` function that binds every port to its adapter. `main.dart` became `Future<void> main() async { WidgetsFlutterBinding.ensureInitialized(); await setupDI(); runApp(...); }`. Screens now resolve deps as `final _api = getIt<IAuthRepository>();`. The `USE_DEV_LEVELS` compile-time flag lives in `setupDI` (registers `DevLevelAdapter` under `ILevelRepository` instead of `LevelHttpAdapter`), removing the runtime toggle previously scattered across screens.
- **F5a — Auth use cases + tests**: `SignInUseCase`, `RegisterUseCase`, `SignOutUseCase`, `RestoreSessionUseCase` in `lib/application/usecases/auth/` — each takes ports by constructor and exposes a `call()` method. Fake-driven tests (no mockito): tiny manual `_FakeAuthRepository`, `_FakeTokenStorage` per test file, AAA + `should_x_when_y` naming. `RestoreSessionUseCase` takes an injectable `now` so expiry logic is deterministically testable. 9 new tests.
- **F5b — Presentation invokes auth use cases**: screens resolve `SignInUseCase` etc. via `getIt` and call them (`await _signIn(email: ..., password: ...)`) — a single async operation from the screen's perspective instead of the two-step "call API then persist" it was doing. `LoginScreen` and `RegisterScreen` no longer import any port directly. `AuthGate` uses `RestoreSessionUseCase` for the "do I have a valid session on disk?" question. `AuthGuard.signOut()` delegates to `SignOutUseCase`. Logout in `LevelsScreen` also switched.
- **F6a — Level + progress use cases + tests**: `GetLevelsUseCase`, `GetStarsByLevelUseCase`, `SubmitLevelResultUseCase`, and the composite `LoadLevelsCatalogUseCase` that fires levels + stars in parallel and tolerates a failed stars fetch by returning an empty map (the policy that used to live inside `LevelsScreen._load()` is now in application). Value object `LevelsCatalog` is defined in the same file as its owning use case. 9 new tests, one specifically covering that the composite propagates a level-repo error but swallows a stars error.
- **F6b — Presentation consumes level/progress use cases**: `LevelsScreen` collapses two futures into a single `_loadCatalog()` call; the private `_LevelsData` bundle class dies (`LevelsCatalog` replaces it). `GameScreen` swaps `_progressRepo.submitScore(...)` for `_submitResult(...)`. Sanity grep of `domain/ports` under `lib/presentation` returned empty — screens depend only on use cases now.
- **F7, F8 — deliberately skipped**: extracting game logic into an `ActivateArrowUseCase` and introducing a sealed `Failure` hierarchy were both in the original plan. Both cut for marginal ROI: `GameSession` is already a well-encapsulated domain aggregate (extracting it would be a wrapper without port coordination), and sealed Failure would touch every screen's catch blocks without adding architecture the professor could point to. Decision recorded in `report.txt` §5.
- **F9 — Verification + merge to main**: grep sanity confirmed the isolation contract — `domain/` and `application/` free of flutter/http/shared_preferences imports; `presentation/` free of `infrastructure/adapters/*` imports except the shared `api_exception.dart` (documented as an accepted exception for the shared exception contract). Merge to `main` with `--no-ff` and a paragraph message describing the whole refactor.

Final counts at merge: tests 60 → 78, analyzer 0 issues, 47 files touched.

**Files affected**:
- **Ports (new)**: `lib/domain/ports/{auth_repository, auth_token_storage, level_repository, progress_repository}.dart`
- **Domain entities (new)**: `lib/domain/models/{level, user_session}.dart`
- **Infrastructure adapters (new)**: `lib/infrastructure/adapters/http/{auth, level, progress}_http_adapter.dart`, `lib/infrastructure/adapters/local/{shared_prefs_token_storage, dev_level_adapter}.dart`
- **DTOs (new)**: `lib/infrastructure/dto/{auth_response_dto, level_dto}.dart`
- **DI (new)**: `lib/core/di/service_locator.dart`
- **Application use cases (new)**: 8 files under `lib/application/usecases/{auth, level, progress}/`
- **Presentation rewritten**: `main.dart`, `auth_guard.dart`, `auth_gate.dart`, `login_screen.dart`, `register_screen.dart`, `levels_screen.dart`, `game_screen.dart`
- **Deleted**: `lib/data/` (9 files, 368 lines)
- **Tests moved**: `test/data/dev_level_source_test.dart` → `test/infrastructure/adapters/local/dev_level_adapter_test.dart` (via `git mv`)
- **Tests added**: 8 use case test files under `test/application/usecases/{auth, level, progress}/`
- **Config**: `pubspec.yaml` (+`get_it`), `pubspec.lock`

**Modifications made by the developer**:
- Insisted on `report.txt` first (a design document capturing target architecture + phase-by-phase plan + risks) BEFORE any code was written. Anchored the refactor in a written contract instead of ad-hoc decisions, and gave a reference to check against when scope creep tempted.
- Enforced per-commit `flutter analyze` + `flutter test` gates. Every commit landed green before the next one started. When F4's registration produced an `unintended_html_in_doc_comment` warning on the `getIt<T>()` mention in a doc comment, fixed it with backticks before committing rather than shipping "0 issues" with a lint noise.
- Skipped F1 (empty scaffold commit) after evaluating that it added git-noise without value. Deviation from the plan, documented in the commit sequence.
- Split each large phase (F3, F5, F6) into `a`/`b` sub-commits: create new files first (safe, tests untouched), then update consumers (invasive, tests must stay green). This gave a rollback boundary if the consumer sweep went wrong.
- Held the line on `api_exception.dart` as a shared contract even though it lives in `infrastructure/adapters/http/`. Screens still catch `on ApiException catch (e)`. Decision documented in the F9 sanity grep output as an accepted exception; rewriting to a `Failure` hierarchy was cut with F8.
- Provided every batch as a copy-paste terminal script that also ran the verify + commit at the end. Zero back-and-forth on file paths and Claude never wrote files directly to the repo (the developer keeps git-author identity clean and no unsolicited edits).

**Lessons learned**:
- **`report.txt` before commit 1 is worth 30 minutes**. Writing down the target architecture, gap analysis, phase plan and risk register before touching code kept every micro-decision anchored. When a mid-refactor temptation arose ("should I also introduce Provider/Riverpod?") the answer came from the report ("no — scope frozen") rather than from an ad-hoc negotiation.
- **Skipping empty-scaffold commits is fine**. Git tracks files, not folders. When F2 materialised the folders with real files, F1 became noise. Skipped, documented in the sequence, no regret.
- **Sub-splitting invasive commits (F3a/b/c, F5a/b, F6a/b) is a rollback insurance policy**. Each sub-commit was green on its own. If F5b's screen updates had introduced a bug, revert wouldn't roll back F5a's use-case creation. Cheap discipline, expensive to lack.
- **Ports + adapters + use cases + DI is the professor's rubric surface**. Four items a grader can point to in the code. F7 (extract game logic) and F8 (sealed Failure) would have added lines without adding surface. Ruthlessly cut both.
- **`api_exception` as shared contract is a pragmatic exception to hexagonal purity**. The alternative (screens wrapping every use case call to translate exceptions into a sealed Failure) is 100+ lines of ceremony to satisfy a rule the professor is unlikely to grade at that level of nuance. Documented once, not litigated further.

## Entry 3 — App game feel features: zoom, haptics/audio, hint/grid, wallet, lives

**Date**: 2026-07-13
**Tool**: Claude Opus 4.7 (via Cowork desktop app)
**Task**: With the hexagonal skeleton merged, the remaining rubric items from the professor's project brief were the game-feel features listed in the PDF: pinch-zoom for the board, haptic + audio feedback, hint and grid-highlight power-ups backed by a coin wallet, and a global lives system that closes the earn-spend loop. All five needed to land in sequence, each as its own atomic feature branch, without breaking the 78 baseline tests.

**Prompt (paraphrased)**: I asked Claude to prioritise by ROI: cheapest win first (zoom = an `InteractiveViewer` wrap), then platform ports (haptics + audio) that introduce a real multi-port use case (`GameFeedbackUseCase`), then the economy (wallet + inventory + power-ups) which touches domain and application heavily, then lives which closes the coin-purchase loop. All local — backend integration for shop/wallet/lives skipped as scope; SharedPreferences adapters implement the same ports so a server-backed adapter can drop in later.

**Result**: Four merged feature branches, 96 tests green (was 78), 0 analyzer issues throughout. Each feature shipped in one or two atomic commits + a `--no-ff` merge to `main`.

- **Zoom + pan (branch `feat/zoom-board`, 1 commit)**: wrapped the `AspectRatio > Stack` in `InteractiveViewer(minScale: 1.0, maxScale: 3.5)` inside `game_screen.dart`. A `TransformationController` field lets a "reset zoom" `IconButton` in the AppBar restore identity. Cell taps propagate correctly because `InteractiveViewer` transforms coordinates back to the child's un-zoomed space. No domain changes.
- **Haptics + audio (branch `feat/haptics-audio`, 1 commit)**: introduced `IHapticsService` and `IAudioService` ports in `lib/domain/ports/`, `FlutterHapticsAdapter` (uses `HapticFeedback.lightImpact/heavyImpact/mediumImpact`) and `SystemSoundsAudioAdapter` (uses `SystemSound.play(SystemSoundType.click/alert)`). Audio adapter deliberately avoided the `audioplayers` package — SystemSound requires no assets and no platform config; a richer adapter drops in later behind the same port. Application-level `GameFeedbackUseCase` composes both ports and exposes semantic methods (`arrowActivated`, `arrowBlocked`, `levelCleared`, `levelFailed`), each firing haptics + audio via `Future.wait`. This is the first use case in the app that composes two service ports — the pattern's actual point of leverage. `game_screen._onCellTapped` fires the right method via `unawaited(...)` so it never blocks the UI. 4 new tests with fake ports counting method invocations.
- **Power-ups + wallet (branch `feat/power-ups`, 2 commits)**: two ports — `IWalletService` (`getBalance/credit/tryDebit`) and `IInventoryService` (`getCount/add/tryConsume(itemId, amount)`) — plus a `PowerUpItems` whitelist domain model (string constants `HINT`, `GRID_HIGHLIGHT`, no enums per project rule). Two adapters: `SharedPrefsWalletAdapter` (seeds first read with 100 coins, atomic `tryDebit`) and `SharedPrefsInventoryAdapter` (seeds first read with 5 hints + 5 grids, key-per-item pattern `inventory_<ID>`). Domain service `ArrowRayCalculator` — pure, computes the cell list an arrow's ray would traverse (same walk as `GameSession.isActivatable` but returning cells instead of bool). Four use cases: `GetWalletBalanceUseCase`, `AwardCoinsForLevelUseCase` (10 base + 5*stars formula), `RevealHintUseCase` (takes activatable arrow IDs and Random?, picks one and consumes 1 hint), `UseGridHighlightUseCase` (takes Board + arrowId, consumes 1 highlight, returns ray cells via the calculator). UI in `game_screen`: coin badge in AppBar, two `FilledButton.tonalIcon` in a power-up bar (Hint (N), Grid (N)) — auto-disabled when count reaches 0. Grid mode is a toggle: when ON, the next arrow tap runs the highlight instead of activating. Highlights self-clear after 3 seconds via a shared `_powerUpTimer`. Cell highlight resolution: blocked flash > hint amber > grid ray cyan. 8 new tests including negative paths (out of hints, out of stock).
- **Lives system (branch `feat/lives-system`, 2 commits)**: `ILivesService` port with `LivesState` value object (`current, max, isEmpty, isFull`). `SharedPrefsLivesAdapter` seeds max at 5, `tryConsume` decrements to 0 minimum, `add` clamps to max. Three use cases: `GetLivesUseCase`, `ConsumeLifeUseCase`, and `PurchaseLifeUseCase` — the last one is the coin-loop closer: reads state, checks not-full, `wallet.tryDebit(20)`, then `lives.add(1)`. Refuses gracefully if wallet insufficient or already at max. UI: hearts + count in AppBar of both `LevelsScreen` and `GameScreen`; "Buy 1 life (20 coins)" button appears in `LevelsScreen` AppBar only when lives < max. `LevelsScreen` blocks entering a level with a snackbar when lives = 0. `GameScreen` consumes 1 life on failure (in `_onCellTapped` when `_session.isFailed` transitions) and on `dispose()` if the player left mid-run without clearing — guarded by `_lifeConsumedThisRun` so a failure + back-tap doesn't double-consume. Per-session `_StatusBar` label renamed "Lives" → "Attempts" to avoid confusion with the global hearts. 6 new tests, including `should_return_false_when_at_max_lives` and `should_return_false_when_wallet_insufficient` for `PurchaseLifeUseCase`.

**Files affected**:
- **Domain (new)**: `ports/{haptics_service, audio_service, wallet_service, inventory_service, lives_service}.dart`, `models/{power_up_items, lives_state}.dart`, `services/arrow_ray_calculator.dart`
- **Infrastructure (new)**: `adapters/platform/{flutter_haptics_adapter, system_sounds_audio_adapter}.dart`, `adapters/local/{shared_prefs_wallet_adapter, shared_prefs_inventory_adapter, shared_prefs_lives_adapter}.dart`
- **Application use cases (new)**: `usecases/game/{game_feedback, reveal_hint, use_grid_highlight}_usecase.dart`, `usecases/wallet/{get_wallet_balance, award_coins_for_level}_usecase.dart`, `usecases/lives/{get_lives, consume_life, purchase_life}_usecase.dart`
- **Presentation modified**: `screens/game_screen.dart` (largest rewrite: zoom, power-up bar, highlight overlays, global-lives AppBar chip, wire all new use cases), `screens/levels_screen.dart` (hearts chip, buy-life button, entry gate)
- **DI**: `service_locator.dart` (+8 bindings across the branches)
- **Tests added**: `game_feedback_usecase_test`, `reveal_hint_usecase_test`, `use_grid_highlight_usecase_test`, `award_coins_for_level_usecase_test`, `get_wallet_balance_usecase_test`, `get_lives_usecase_test`, `consume_life_usecase_test`, `purchase_life_usecase_test` (18 tests total across all four branches)

**Modifications made by the developer**:
- Locked the "local adapter first, server later" pattern for shop/wallet/inventory/lives explicitly. Rationale: the professor's rubric weighs the port pattern and the earn-spend loop, not whether balances survive an app reinstall. `SharedPrefs*Adapter` classes all implement the same ports a future HTTP adapter would, so the seams are already in place.
- Kept `PowerUpItems` as a plain class with `static const String` members instead of an enum (professor rule). Adapter's `_defaults` map uses those constants as compile-time keys — works because `PowerUpItems.hint = 'HINT'` is a valid const expression.
- Renamed the per-session attempt counter "Lives" → "Attempts" in the game screen's `_StatusBar` after the double meaning surfaced during the lives commit. The global hearts are lives; the per-run counter is attempts. Avoids "why did I lose 3 lives from one level" during demo.
- Guarded double-consumption in `game_screen` with `_lifeConsumedThisRun`. Both the failure path and the dispose path check it before consuming. `dispose()` fires the async consume without awaiting because the widget is being torn down; the levels screen refreshes lives on return, so the UI stays consistent.
- Used the `SystemSound` adapter over `audioplayers` deliberately — no assets to ship, no iOS Info.plist / Android permissions to add, no plugin channel to debug. Documented the limitation ("simple click sound") in the adapter docstring so a future improvement is cheap.
- Ran the game_screen's async feedback calls as `unawaited(_feedback.arrowActivated())`. Fire-and-forget is correct here because there's nothing meaningful to return, and awaiting would slow the tap response. `unawaited` from `dart:async` silences the lint the plain call would trigger.

**Lessons learned**:
- **First real coordination use case is worth waiting for**. Auth/level/progress use cases were wrappers over single ports. `GameFeedbackUseCase` composes `IHapticsService + IAudioService`, `PurchaseLifeUseCase` composes `IWalletService + ILivesService`. These are the ones that justify the pattern — the professor sees "orchestrator over multiple ports" and there's non-trivial logic to point at.
- **String-constant whitelists in a class are fine as long as adapters use them**. `PowerUpItems.hint` is compile-time const, safe as a map key, one grep away for renames. No enum needed, no ceremony added.
- **Local persistence adapters buy you the same seams as server ones**. `SharedPrefsWalletAdapter` and a future `HttpWalletAdapter` share the port. Use cases and presentation don't know or care. Migration is a single-line change in `setupDI` when the backend endpoint is ready.
- **Double-consume guards are 1-line, missing them is 30 minutes of debug**. `_lifeConsumedThisRun` was cheap; without it, "player fails a level" would consume 1 life at the fail branch AND another at dispose. Guarded once, correct forever.
- **`unawaited` is the right tool when the future's result is intentionally ignored**. Silences `unawaited_futures` lint without hiding actual mistakes. Combined with fire-and-forget UI feedback, it keeps the tap response instant.

## Entry 4 — App: generate-more FAB and star-bug fix (real timeMs + server refetch)

**Date**: 2026-07-13
**Tool**: Claude Opus 4.7 (via Cowork desktop app)
**Task**: After Entries 2-3 shipped the hexagonal skeleton and game-feel features, playtest surfaced two gaps. First, the backend's `POST /api/levels/generate` endpoint existed but the app had no UI to hit it — a full-stack integration that only worked from `curl`. Second, the win dialog always displayed "Stars earned: * * *" (3 stars) even when the player had spent multiple attempts on blocked taps, because the app hardcoded `timeMs: 0` in the score submission — server's `starsFor(0)` returned 3 unconditionally.

**Prompt (paraphrased)**: I asked Claude to (a) wire the "Generate more levels" button as an atomic feature branch (FAB + bottom sheet + use case + port method), and (b) find the source of the always-3-stars bug and fix it end-to-end so the dialog matches what the server records.

**Result**: Two feature branches, both merged to main.

- **`merge: generate more levels on demand from the app`** (commit `8c6a97d`): extended `ILevelRepository` with `generate({required String difficulty})`; `LevelHttpAdapter` POSTs to `/levels/generate` and parses the returned level via existing `LevelDto.fromJson().toDomain()`; `DevLevelAdapter` throws `UnsupportedError` (offline fixture has nothing to generate against). New use case `GenerateLevelUseCase` — a thin wrapper that delegates to the port. Registered in DI. In `LevelsScreen`: added a `FloatingActionButton.extended` labeled "Generate level" that opens a `showModalBottomSheet` with three `ListTile` options (EASY/MEDIUM/HARD); tapping one invokes the use case, shows a loading indicator on the FAB, and calls `_reload()` on success so the new level appears at the bottom of the catalog. Two fake-driven tests cover argument passthrough and error propagation. Updated the fakes in `get_levels_usecase_test` and `load_levels_catalog_usecase_test` to include the new stub method (Python inline patcher, asserts on match count so a silent no-op fails the batch). Test total went from 96 to 98.

- **`merge: fix always-3-stars bug and align dialog with server`** (commit under branch `fix/real-timems-and-server-stars`): added `DateTime? _sessionStartTime` field in `_GameScreenState`, set in `_startSession()`, used by a new `_elapsedMs()` helper. `_submitAndShowWin()` now passes `elapsedMs` to `_submitResult` and, on success, refetches stars via `GetStarsByLevelUseCase` to get the server's decision. `_serverStars` field stores that value; the dialog reads `_serverStars ?? _session.starsEarned` and displays both the actual elapsed seconds and the star count. `_coinsEarnedThisRun` now credits based on the server's stars (not the client's optimistic estimate), so the wallet and the dialog agree. No app tests changed — the fix is entirely inside the state-managing methods, no domain contract shift.

Together with the backend's parallel fix (`stars = min(movesBased, timeBased)`, see backend AI_USAGE Entry 27), the loop now behaves correctly end-to-end: a fast clean run shows 3 stars in the dialog AND in the levels list; a fast run with two blocked taps shows 1 star everywhere.

**Files affected**:
- **Domain**: `lib/domain/ports/level_repository.dart` (added `generate` method)
- **Infrastructure**: `lib/infrastructure/adapters/http/level_http_adapter.dart` (POST /generate), `lib/infrastructure/adapters/local/dev_level_adapter.dart` (throws)
- **Application**: `lib/application/usecases/level/generate_level_usecase.dart` (new)
- **Presentation**: `lib/presentation/screens/levels_screen.dart` (FAB + bottom sheet + reload), `lib/presentation/screens/game_screen.dart` (session start time, elapsed ms, server-stars refetch, dialog display)
- **DI**: `lib/core/di/service_locator.dart` (added generate use case)
- **Tests**: `test/application/usecases/level/generate_level_usecase_test.dart` (new, 2 tests), `test/application/usecases/level/get_levels_usecase_test.dart` (added stub for new port method), `test/application/usecases/level/load_levels_catalog_usecase_test.dart` (added stub)

**Modifications made by the developer**:
- Kept `_session.starsEarned` (client's attempts-based estimate) as the fallback in the dialog display, so if the server refetch fails the player still sees a reasonable number instead of a "-". The `??` operator makes the fallback silent.
- Held on adding `IBoardGenerator` as an abstraction layer between the use case and the concrete `RandomBoardGenerator` on the backend side; the app-side `GenerateLevelUseCase` correctly stays as a thin wrapper over `ILevelRepository.generate` because the app doesn't own the generator — the server does. This is the correct hexagonal shape: the port abstracts the REMOTE capability.
- Documented in `DevLevelAdapter.generate`'s throw message that offline fixture cannot fulfil generation, so a future contributor toggling `USE_DEV_LEVELS=true` and hitting the FAB gets a legible error instead of a silent hang.
- Ran cold restart (`flutter clean && flutter run`) after the fix because Flutter's hot restart doesn't rerun `setupDI()` — the app was seeing cached bindings. Documented in the manual test recipe.

**Lessons learned**:
- **A hardcoded `0` in a submitted field is invisible when the receiving side has a "0 is fine" fallback**. `timeMs: 0` was written in the initial `_submitAndShowWin` as a placeholder ("we'll fix this later"); the backend's `starsFor(0)` returned 3 (the fastest bucket), so every run recorded a max score and the placeholder shipped. The moment you hardcode a magic value at a seam, it becomes production behavior; either compute the real value or throw so the seam fails loudly.
- **Server-side refetch is the right way to keep dialog stars in sync with list stars**. Trying to duplicate the server's grading formula on the client would drift the moment either side changed. Refetching after submit costs one round-trip and guarantees the two surfaces show the same number.
- **A FAB is worth more than a menu entry when the action is the one the player will want most often**. The bottom-sheet difficulty picker is one extra tap, but it makes the difficulty choice explicit — critical when EASY, MEDIUM, and HARD produce visibly different boards. Skipping the picker and defaulting to "surprise me" would have hidden the generator's variety.
- **Cold restart matters after DI changes**. `flutter clean && flutter pub get && flutter run` reset the isolate and re-ran `setupDI`. Any change to `service_locator.dart` — new binding, changed adapter — requires this dance because `get_it` registrations happen once at start and hot restart doesn't clear the singleton state.
